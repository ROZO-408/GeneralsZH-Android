# 2026-07 Development Diary

## 2026-07-06 — Android tablet port, scaffolding landed

Started the Android tablet port today. The repo already ships a real iOS/iPad
port (~2,200 lines, fully playable per the playbook), so the smart move was to
treat that as the template rather than start from scratch. Research first,
then implementation.

### Research verdict
The single load-bearing question was: **can DXVK (D3D8→Vulkan) run natively on
Android?** It has never been built for Android upstream (doitsujin closed #1183
with "No" — but that was about Wine/emulation, not native linking). The native
non-Wine path is the same one the game already uses on Linux and iOS. Verdict:
**feasible but unproven** — the build blockers are concrete and patchable (gate
`-msse` behind x86; the aarch64 NEON bitops already exist), but the runtime gate
(driver quality) can only be answered on hardware. No Plan B exists, so it must
be spiked before committing the full budget.

Everything else (SDL3, OpenAL/Oboe, touch, deps, packaging) has a known path.
The strongest finding: **11 of the 17 iOS-guarded engine blocks port by simply
widening the `#ifdef`**, including the entire ~230-line touch gesture state
machine — the highest-effort iOS work.

### What landed today
- Introduced a `SAGE_MOBILE` macro (iOS ∨ Android) and widened the platform-
  agnostic mobile code to it: touch gesture machine, lifecycle render-gate,
  high-DPI drawables, `-xres/-yres` aspect matching. iOS-only mechanisms
  (MoltenVK, `funopen`, bundle paths) stay on the narrower `TARGET_OS_IPHONE`
  guard.
- Android bootstrap in `SDL3Main.cpp`: chdir to internal storage (the VFS analog
  of iOS's bundle GameData), portable stderr→file sink via `dup2` (bionic has no
  `funopen`), DXVK cache path.
- DXVK Android build groundwork: `Patches/dxvk-android.patch` (gate `-msse` behind
  x86 + high-DPI WSI fix), `cmake/meson-android-aarch64-cross.ini.in` (NDK
  cross-file), `cmake/dx8.cmake` Android branch (Meson ExternalProject →
  `libdxvk_d3d8.so`/`libdxvk_d3d9.so`).
- `android-vulkan` CMake preset + `arm64-android` vcpkg triplet.
- `android/` Gradle project (build.gradle with `externalNativeBuild` → repo-root
  CMakeLists.txt, AndroidManifest, GameActivity SDLActivity subclass).
- `scripts/build/android/package-android-zh.sh` + `stage-fonts.sh`.
- `PHASE06_ANDROID_PORT.md` and this diary entry.

### Honest status
**Scaffolding complete; runtime unverified.** No APK has been built or run on a
device yet. The DXVK-on-Android runtime spike is the explicit next gate — if a
d3d8 clear renders via DXVK on a real Android Vulkan driver, the port is viable
and the rest is performance/packaging work. If the driver can't give DXVK the
extensions it queries, the port is blocked (no fallback translator exists).

The Android tooling is all present locally (NDK r27, SDK API 35, an arm64-v8a
AVD, JDK 17, Gradle), so the spike can run as soon as there's a device or the
emulator's Vulkan support is confirmed.
